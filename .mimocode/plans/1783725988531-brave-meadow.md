# Ground-Up Networking Fix Plan

## Root Cause: v0.33.0 Regressions

Comparing the **stable backup** (`09_07_26_1920`) with the **current HEAD** reveals that commit `defab10` (v0.33.0) introduced several regressions that broke file transfers. The GitHub source (`origin/main`) has the same regressions.

### Critical Regressions (must fix)

| # | What changed | Stable (working) | Current (broken) | Impact |
|---|-------------|-----------------|------------------|--------|
| 1 | **Chunk size for relay** | `64KB` | `256KB` | 4x larger chunks overwhelm relay circuits |
| 2 | **Relay pacing** | `50ms` | `20ms` | Combined with #1, 10x throughput demand on relay |
| 3 | **Stall watchdog timer** | `t.last_update` | `t.last_retry` | Stale transfers not refreshed, evicted prematurely |
| 4 | **`is_relayed` default** | `unwrap_or(true)` | `unwrap_or(false)` | Unclassified peers assumed direct, wrong params |
| 5 | **In-flight limit** | `4` (relay) / `8` (direct) | `8` always | Doubles pressure on relay circuits |

### Secondary Issues

| # | Issue | Status |
|---|-------|--------|
| 6 | Clear chat not working | **Already fixed** (`clear_group_chat` FFI added) |
| 7 | Messages relayed on same network | **Already fixed** (mDNS override + force-disconnect) |

## Summary

The core problem is that v0.33.0 optimized for throughput (bigger chunks, faster pacing, higher concurrency) but the relay circuits can't handle it. The stable backup used conservative relay parameters that worked reliably. The fix is to **restore the relay-specific parameters from the stable backup** while keeping all the structural improvements.

Data rate comparison:
- **Stable**: 64KB / 50ms = **1.28 MB/s** through relay
- **Current**: 256KB / 20ms = **12.8 MB/s** through relay (10x overload)
