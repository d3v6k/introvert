# Fix Relay Connection Flapping — Restore Stable Reservation Pattern

**Date:** 2026-07-06
**Status:** Ready for implementation
**Scope:** 2 files, ~12 lines changed

---

## Root Cause

The stable backup (`stable_0.1.0_2026_07_06`) had `relay_reservations.insert()` at **6 locations** — every `listen_on()` call site. My relay stability changes removed all 6 and added only 1 insert at `ReservationReqAccepted` time.

This created a gap: between `listen_on()` (reservation requested) and `ReservationReqAccepted` (reservation accepted), `relay_reservations` is empty. During this gap, if `ConnectionClosed` fires (RBN connection drops briefly), the reservation is cleared and never re-inserted. The 15s status check sees `relay_reservations` is empty → status 4 (CONNECTING) → triggers VPN stale detection → force-disconnects all RBNs → infinite loop.

**The stable version worked because** `relay_reservations` was always populated at `listen_on()` time, so even if `ConnectionClosed` cleared it, the next `listen_on()` call (from the 15s recovery tick) immediately re-inserted it.

---

## Fix

Re-add `relay_reservations.insert()` at all 6 original `listen_on()` call sites. Keep the insert at `ReservationReqAccepted` as a confirmation. This restores the stable behavior where `relay_reservations` is always non-empty when a reservation is requested.

**Files:**
- `src/network/mod.rs` — 6 locations
- `for_linux/src/network/mod.rs` — 2 locations (RBN has fewer call sites)

### Locations to restore (client `src/network/mod.rs`)

1. **Line 288** — Startup reservation request
2. **Line 630** — Status check re-request (Step 1)
3. **Line 739** — Fast reconnect
4. **Line 1420** — Identify event (relay-capable peer discovered)
5. **Line 2135** — ConnectionEstablished with RBN
6. **Line 2225** — ListenerClosed recovery

### Locations to restore (RBN `for_linux/src/network/mod.rs`)

1. **Line 1514** — Identify event
2. **Line 2158** — ConnectionEstablished

### Pattern

At each `listen_on()` site, after `self.swarm.listen_on(relay_addr)` returns `Ok(id)`:
```rust
self.relay_reservations.insert(peer_id);  // Re-add this line
self.relay_listeners.insert(id, peer_id);
```

### Keep existing code

- Keep `relay_reservations.insert(relay_peer_id)` in `ReservationReqAccepted` (confirmation)
- Keep `relay_listeners.retain()` in `ConnectionClosed` (clean up stale listeners)
- Keep `relay_reservations.is_empty() && relay_listeners.is_empty()` check in VPN stale detection
- Keep `has_confirmed_reservation` in status check
