## VPN Message Failure тАФ Root Cause Analysis

### Comparison: v54 vs Current VPN/Mobile Handling

| Aspect | v54 | Current |
|--------|-----|---------|
| **Tunnel WebSocket URL** | `ws://47.89.252.80:80/tunnel` (plaintext, port 80) | `wss://47.89.252.80/tunnel` (TLS, port 443) |
| **VPN detection from Flutter** | Dropped (no Rust handler) | Handled, activates tunnel |
| **VPN bootstrap isolation** | Keeps public IPs alongside tunnel | Clears all, isolates to tunnel-only |
| **Tunnel stale detection** | None | 60s age check + 15s/30s force-reset |
| **fast_reconnect_interval** | Does not exist | 5s aggressive reconnect |
| **Mobile data handling** | None | Mailbox skip, pacing, IPv6 |

### Root Cause
The current code uses `wss://` on port 443 for the WebSocket tunnel. Many VPN configurations block or interfere with TLS WebSocket on port 443. The v54 backup used `ws://` on port 80 (plaintext) which bypasses most VPN restrictions.

### Fix Required
**File:** `src/network/types.rs` line 8
**Change:** Add `ws://` port 80 fallback when VPN is detected.

Current:
```rust
pub const RBN_WS_URL: &str = "wss://47.89.252.80/tunnel";
```

Fix: Add a plaintext fallback constant and use it when `connectivity_type == 5`:
```rust
pub const RBN_WS_URL: &str = "wss://47.89.252.80/tunnel";
pub const RBN_WS_URL_PLAIN: &str = "ws://47.89.252.80:80/tunnel";
```

Then in `ActivateTunnel` handler and startup tunnel activation, use `RBN_WS_URL_PLAIN` when on VPN (type 5), and `RBN_WS_URL` otherwise.
