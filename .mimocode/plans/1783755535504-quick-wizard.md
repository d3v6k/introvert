# Plan: Lazy File Transfer Architecture

## Problem
File transfers persist across app restarts via SQLite (`pending_file_chunks` table) and RAM (`incoming_transfers`, `pending_messages`). Stale transfers from previous sessions clog the network loop, starve the command channel, and cause the orphan group spam to block real file chunk delivery. The current "eager resume" model tries to resume ALL pending transfers on startup, which is fragile and resource-intensive.

## Proposed Architecture
**"Flush on start, sync on open, pull on demand"**

1. **App start**: Clear all pending file transfers (RAM + SQLite). Only text messages survive in the outbox.
2. **Chat opened**: Chat sync runs → discovers missing `[FILE]:` manifests → shows placeholder UI → triggers `start_pull` for each missing file.
3. **Files download**: Through the normal relay/direct path, with progress shown in the chat.

## Changes

### 1. Rust: Clear file transfer state on engine start
**File:** `src/network/mod.rs` (~line 175, after swarm initialization)

Add cleanup logic:
```
- Clear `pending_file_chunks` table via storage.cleanup_old_pending_chunks() or a new truncate function
- Clear `incoming_transfers` HashMap (will be empty on fresh start anyway)
- Clear `pending_messages` entries that are FileChunk/FileChunkRequest
- Keep `outbox` table intact (text messages)
```

**File:** `src/storage.rs`
- Add `truncate_pending_file_chunks()` — DELETE all rows from `pending_file_chunks`
- Keep `outbox` table untouched

### 2. Rust: Add FFI for clearing stale transfers
**File:** `src/lib.rs`
- Export `introvert_network_flush_stale_transfers()` — called by Flutter on app launch after engine start
- Clears `incoming_transfers`, `pending_messages` file payloads, and `pending_file_chunks` table

### 3. Flutter: Call flush on app launch
**File:** `lib/src/ui/main_shell.dart` or initialization sequence
- After engine start + network start, call `_client.flushStaleTransfers()`
- This ensures clean slate before any chat sync happens

### 4. Flutter: Auto-download all missing files after chat sync
**File:** `lib/views/chat_screen.dart` (`initState` ~line 787)
- After `syncChatMessages()` completes and `_loadMessages()` runs:
  - Scan loaded messages for `[FILE]:` entries where local file doesn't exist
  - For each missing file, call `startPull()` silently (respecting `_pullRequested` dedup)
  - Show `FileTransferBubble` with progress indicator — no user action needed
  - All files auto-download in background

**File:** `lib/views/group_chat_screen.dart` (`initState` ~line 212)
- Same pattern: after sync + reload, scan for missing `[FILE]:` entries
- Auto-trigger `start_pull` for ALL missing files silently

### 5. Flutter: Remove eager pull from message loading
**Files:** `chat_screen.dart`, `group_chat_screen.dart`
- Remove `start_pull` calls from `_loadMessages()` / `_reloadMessages()` (the current auto-pull on every message load)
- Replace with: only pull after chat sync completes, or on user tap

### 6. Cleanup: Remove stale transfer eviction logic
**File:** `src/network/mod.rs`
- The stall watchdog and stale eviction logic (lines ~454-543) can be simplified since transfers won't persist across restarts
- Keep the watchdog for active in-session transfers, but remove the 300s queued transfer eviction (no longer needed)

## Verification
1. Build all 3 platforms: `make all`
2. Start Android on VPN, Mac/iOS on WiFi
3. Send text message from Android → should arrive on Mac/iOS (outbox works)
4. Send file from Android to group → Mac/iOS receive `[FILE]:` manifest via gossipsub
5. Open group chat on Mac → sync runs → all missing files auto-download silently with progress
6. File transfer should complete through relay circuit
7. Restart Android → no stale transfers in logs, clean startup, no orphan group spam
8. Re-open group chat on Mac → re-syncs, discovers any new missing files, auto-downloads them
9. Verify: no `Stall retry` spam, no `InboundCircuit DB flush` loops, no `forward_to_mesh` starvation
