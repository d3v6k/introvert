# Comprehensive Codebase Audit â€” Introvert

## Executive Summary

Deep audit of the Rust networking core, FFI layer, and file transfer pipeline. Found **3 critical bugs**, **5 high-severity issues**, and **14 medium-severity issues**. The Android OOM crash is caused by unbounded memory growth in `early_chunks`, `incoming_transfers`, and `outbound_tracker`, combined with excessive `FileChunk` payload cloning.

---

## CRITICAL Issues (Fix Immediately)

### C1: `total_chunks: 0` in DB Flush Path Corrupts Receiver State

**File:** `src/network/mod.rs` lines 1308, 2431, 2475

When flushing pending DB chunks, `total_chunks` is hardcoded to `0`:
```rust
let payload = SignalingPayload::FileChunk {
    transfer_id, chunk_index,
    total_chunks: 0,  // BUG â€” corrupts receiver
    data_base64,
};
```
In `handle_file_chunk` (line 1520): `transfer.total_chunks = total_chunks` overwrites the real value with `0`, so `is_complete` (line 1535: `received_chunks.len() == total_chunks as usize`) is never true. The transfer can never complete.

**Fix:** Store `total_chunks` in the `pending_file_chunks` table, or look it up from `incoming_transfers`/`active_seeders` when re-encoding.

### C2: `dequeue_pending_chunks` Never Deletes Rows

**File:** `src/storage.rs` line 1256-1269, `src/network/mod.rs` line 1296-1317

`dequeue_pending_chunks` is a SELECT â€” it reads rows but never deletes them. The same chunks are re-forwarded every 30 seconds for up to 24 hours (only `cleanup_stale_pending_chunks(86400)` removes them).

**Fix:** After successful send in the periodic flush, delete the chunk from the DB. Or increment `retry_count` and skip after max retries.

### C3: `early_chunks` Can Hold Up to 5GB

**File:** `src/network/mod.rs` lines 1791-1808

`early_chunks` caps at 100 transfers Ă— 1000 chunks Ă— 50MB per transfer = **5GB** of base64-encoded data. No TTL-based eviction.

**Fix:** Reduce total cap to ~50MB across all transfers. Add timestamp and evict entries older than 60s.

## HIGH Issues

### H1: `outbound_tracker` Has No Cap â€” Holds Full Payloads

**File:** `src/network/service.rs` line 44, `src/network/mod.rs` various

`outbound_tracker: HashMap<OutboundRequestId, (PeerId, SignalingPayload)>` stores the FULL payload (including `data_base64` for FileChunk) for every in-flight request. No cap, no periodic cleanup. Lost requests leak permanently.

**Fix:** Add global cap (e.g., 500 entries). Add periodic cleanup in heartbeat for entries older than 30s. Store only a payload reference/ID, not the full payload.

### H2: FileChunk Payloads Cloned 3-5 Times in `forward_to_mesh`

**File:** `src/network/mod.rs` lines 3850, 3883, 3956, 3961, 3980, 4067, 4096, 5055, 5057, 5059

Each `SignalingPayload::FileChunk` (up to 87KB base64) is cloned multiple times: once for `pending_messages`, once for `outbound_tracker`, once for the wire send. A single chunk can be cloned 3-5 times.

**Fix:** Use `Box<SignalingPayload>` or pass by ownership. The `outbound_tracker` should store a lightweight reference, not the full payload.

### H3: Gossipsub Topics Subscribed But Never Unsubscribed

**File:** `src/network/mod.rs` lines 187, 668, 3937, 5035

Zero `gossipsub.unsubscribe()` calls in the entire codebase. `file-transfer-global` topic causes ALL nodes to receive ALL file chunk traffic. Topic subscriptions grow monotonically.

**Fix:** Unsubscribe from `file-transfer-{group_id}` when leaving a group. Replace `file-transfer-global` with transfer-specific topics. Add periodic cleanup of stale topic subscriptions.

### H4: `incoming_transfers` No Byte-Level Memory Limit

**File:** `src/network/mod.rs` line 7448

Each `IncomingTransfer` holds `received_chunks: HashMap<u32, Vec<u8>>` â€” the entire file decoded in RAM. With 50 concurrent transfers of 100MB files = **5GB**.

**Fix:** Track total bytes across all `incoming_transfers`. Reject new transfers when total exceeds ~200MB. Cap per-transfer at `total_size` from manifest.

### H5: Silent Command Drops Throughout FFI Layer

**File:** `src/lib.rs` â€” 40+ instances of `let _ = tx.send(...)`

All `tx.send(...).await` calls silently discard errors. If the channel is disconnected or full, commands are lost with no feedback to Dart.

**Fix:** Log errors from `tx.send()` via `dispatch_debug_log`. Return error `FfiResult` when `try_send` fails.

## MEDIUM Issues

### M1: `active_seeders` Never Times Out for Group Transfers

**File:** `src/network/mod.rs` lines 8217-8218

Group seeders are intentionally kept alive indefinitely. Over weeks/months, stale entries accumulate and Kademlia advertises the node as a provider for files it may no longer have.

**Fix:** Add periodic cleanup â€” check if `file_path` still exists on disk, remove if not.

### M2: `noise_sessions` Unbounded Growth

**File:** `src/network/service.rs` line 26

No cap, no TTL. Sessions for peers that connect briefly and disappear accumulate.

**Fix:** Add LRU eviction (max 100 sessions) or TTL-based cleanup.

### M3: `relay_reservations` Never Explicitly Removed

**File:** `src/network/service.rs` line 41

When a relay connection drops, the reservation entry is NOT removed. The set grows over time.

**Fix:** Remove from `relay_reservations` in `ConnectionClosed` handler when the peer is the RBN.

### M4: `pending_messages` Global Cap Missing

**File:** `src/network/service.rs` line 28

Per-peer cap exists (50 payloads), but no global cap across all peers. N peers Ă— 50 payloads = unbounded.

**Fix:** Add global cap (e.g., 500 total payloads). Drop oldest when exceeded.

### M5: `CString::new().unwrap()` Panics in Async Tasks

**File:** `src/lib.rs` lines 617-666

If Solana signature or error message contains a null byte, `CString::new` returns `Err`, `.unwrap()` panics, callback never called, Dart hangs indefinitely.

**Fix:** Replace `.unwrap()` with `.unwrap_or_default()` or handle the error.

### M6: `std::sync::Mutex` Poisoning Risk

**File:** `src/network/service.rs` line 73

`active_db_flush_peers` uses `std::sync::Mutex` while everything else uses `parking_lot`. `.lock().unwrap()` at `mod.rs:2408` will panic if poisoned.

**Fix:** Switch to `parking_lot::Mutex` or handle poisoning.

### M7: `inflight_requests` Counter Can Leak

**File:** `src/network/mod.rs` lines 3845-3852, 3967-3969

Counter incremented on send, decremented on response/failure. If response never arrives (timeout, crash), counter drifts upward, eventually hitting the limit and permanently buffering chunks.

**Fix:** Add periodic counter reset tied to `outbound_tracker` entries.

### M8: `FileTransfer` Manifest Uses Unverified `sender_peer_id`

**File:** `src/network/mod.rs` line 7300

The manifest uses `sender_peer_id` from the payload rather than the authenticated `peer` from the transport layer. A malicious relay can forge the sender identity.

**Fix:** Always use `peer` (authenticated transport peer) as seeder identity.

### M9: Blanket `clippy::not_unsafe_ptr_arg_deref` Allow

**File:** `src/lib.rs` line 6

Silences ALL clippy warnings about dereferencing raw pointers in `extern "C"` functions. New functions without null checks compile silently.

**Fix:** Remove blanket allow, add per-function `#[allow(...)]` with justification.

### M10: `shutdown_background()` Doesn't Cancel Tasks

**File:** `src/lib.rs` line 2807

Tasks holding `Arc<StorageService>` references continue running after engine shutdown. Economy monitoring loop runs indefinitely.

**Fix:** Use cancellation token and graceful shutdown with timeout.

### M11: Multiple Tokio Runtimes Created Per Recon Call

**File:** `src/lib.rs` lines 2443, 2473, 2498, 2612, 2660, 2768

Each `intro_claw_run_network_recon` call creates a NEW `tokio::runtime::Runtime`. Wasteful and can cause resource exhaustion.

**Fix:** Use the existing engine runtime instead of creating throwaway runtimes.

### M12: `discovered_anchors` Vec Never Drained

**File:** `src/network/service.rs` line 36

Entries are pushed but never removed. Grows linearly with unique anchors discovered.

**Fix:** Add periodic cleanup in heartbeat.

### M13: `member_count` Never Decremented on Remove

**File:** `src/e2ee.rs` line 220

`remove_member` sets tree node to `Empty` but doesn't decrement `member_count`. Future `tree_size()` calculations are wrong.

**Fix:** Decrement `member_count` in `remove_member`.

### M14: Base64 83% Overhead in Memory

**File:** `src/network/types.rs` line 176

`FileChunk.data_base64: String` stores base64-encoded data (83% overhead). A 64KB chunk becomes 87KB in memory. This is stored in `pending_messages`, `outbound_tracker`, and `early_chunks`.

**Fix:** Consider storing raw bytes and encoding only at the wire boundary. Or use a more compact encoding.
