# Fast Cross-Network File Transfer Plan

**Baseline:** v0.30.2/v0.30.3 (commit `b61c4b5` / `f476ae0`) — mailbox system removed, peer-only outbox with dedup/priority, 50 msg/s batch rate limiting, priority sorting (file chunks first), stagnant message detector.

**Goal:** Fast cross-network group and 1:1 file transfers over relay circuits.

**Current bottleneck:** Relay transfers capped at ~256 KB/s. A 10MB file takes ~40s over relay. Root causes: 64KB relay chunks, 4 in-flight limit, 250ms push pacing, pull-only relay model.

---

## Phase 1: Constant Tuning (No Protocol Changes)

All in `src/network/mod.rs`. Pure constant changes — ship and test same day.

### 1A. Relay chunk size: 64KB → 256KB
- **Lines:** ~L6390, ~L7626
- **Before:** `if final_is_relayed { 64 * 1024 } else { 256 * 1024 }`
- **After:** `256 * 1024` for both
- **Why:** 64KB forces 160 round-trips for 10MB. 256KB = 40 round-trips. Receiver reads `chunk_size` from manifest — backward compatible. Codec supports up to 10MB payloads.
- **Impact:** 4x fewer round-trips.

### 1B. Relay in-flight limit: 4 → 8
- **Line:** ~L3224
- **Before:** `let limit = if is_relayed_conn { 4 } else { 8 };`
- **After:** `let limit = 8;`
- **Why:** 4 × 64KB = 256KB in-flight. 8 × 256KB = 2MB in-flight — reasonable for relay circuits.
- **Impact:** With 256KB chunks and 100ms RTT: 8 × 256KB / 0.1s = 20.5 MB/s theoretical max (vs 2.56 MB/s). **~8x improvement.**

### 1C. Relay push pacing: 250ms → 50ms
- **Line:** ~L7769
- **Before:** `tokio::time::sleep(Duration::from_millis(if is_direct { 20 } else { 250 })).await;`
- **After:** `tokio::time::sleep(Duration::from_millis(if is_direct { 20 } else { 50 })).await;`
- **Why:** 250ms = 4 chunks/sec × 64KB = 256 KB/s. **This is the single biggest bottleneck.** 50ms = 20 chunks/sec × 256KB = 5 MB/s.
- **Impact:** **~20x improvement** for relay push throughput.

### 1D. Relay initial send delay: 2000ms → 500ms
- **Line:** ~L7716
- **Before:** `tokio::time::sleep(Duration::from_millis(if is_relayed { 2000 } else { 200 })).await;`
- **After:** `tokio::time::sleep(Duration::from_millis(if is_relayed { 500 } else { 200 })).await;`
- **Why:** Relay circuit is already established when manifest sends successfully. 2s is excessive.
- **Impact:** Saves 1.5s on every relay transfer startup.

### 1E. Receiver pull pacing: 50ms → 20ms
- **Line:** ~L6394
- **Before:** `let base_pacing = if is_direct_p2p { 10u64 } else { 50 };`
- **After:** `let base_pacing = if is_direct_p2p { 10u64 } else { 20 };`
- **Why:** 50ms between pull requests starves the pipeline. 20ms keeps it saturated.
- **Impact:** Eliminates pipeline stalls in pull mode.

### Phase 1 Combined Impact

| Metric | Before | After | Improvement |
|--------|--------|-------|-------------|
| Relay push throughput | ~256 KB/s | ~5 MB/s | **~20x** |
| Relay pull throughput | ~2.56 MB/s | ~20 MB/s | **~8x** |
| 10MB file via relay push | ~40s | ~2s | **~20x** |
| 10MB file via relay pull | ~4s | ~0.5s | **~8x** |
| Initial latency (relay) | 2s + RTTs | 0.5s + RTTs | 1.5s saved |

## Phase 2: Relay Push Model (Highest Impact)

The current relay model is pull-only: the sender returns early at L7710-7713 and waits for per-chunk FileChunkRequest. Switching to push eliminates one round-trip per chunk.

### 2A. Enable sender-side push for relay connections
- **Lines:** ~L7710-7713
- **Change:** Remove the early return:
  ```rust
  // REMOVE:
  if is_relayed {
      info!("... Relay mode - waiting for chunk requests ...");
      return Ok(());
  }
  ```
- **Why:** The stall watchdog (L464-492) already auto-recovers from stalls by transitioning push→pull (`t.is_relayed = true` at L492). Push is strictly better when the circuit is healthy. The receiver-side `handle_file_chunk` already processes unsolicited chunks (chunks are indexed, duplicates are harmless via `received_chunks` HashMap).
- **Impact:** Eliminates 40 × RTT = 4-8 seconds for a 10MB file.
- **Risk:** Medium. Mitigated by: in-flight limit (8 from 1B), stall watchdog (8s timeout), pacing (50ms from 1C). The v0.30.2 stagnant message detector (120s → ForceMeshRefresh) provides additional safety.
- **Backward compat:** Old receivers already handle unsolicited chunks. No protocol change.

### 2B. Batch initial pull requests (fallback improvement)
- **Lines:** ~L6453-6463
- **Change:** Remove inter-request delay in initial pull burst. Pipeline depth already limits concurrency. The in-flight limit in `forward_to_mesh` (L3224) provides real backpressure.
- **Impact:** Saves 150-750ms on pull startup.

## Phase 3: Group Transfer Optimization

### 3A. Sender-side parallel push with shared file read
- **Lines:** ~L4100-4143 (group SendFile handler)
- **Change:** For group sends, read the file once and fan-out chunks to all members via broadcast channel instead of each member task reading the file independently.
- **Implementation:** 
  1. In group `SendFile` handler, read file once into a `tokio::sync::broadcast` channel
  2. Spawn single reader task that reads chunks and broadcasts
  3. Each per-member push task subscribes and forwards with per-member pacing
- **Impact:** Reduces disk I/O from N reads to 1 read. For 5 group members: saves 40MB of disk reads for a 10MB file.

### 3B. Parallel DHT provider download
- **Lines:** ~L456 (FindProviders in stall watchdog)
- **Change:** When new provider discovered via DHT during group download, immediately dispatch chunk requests to new provider for next pipeline window instead of waiting for stall.
- **Impact:** True parallel download from multiple seeders. 3 seeders = 3x throughput.

## Phase 4: Resilience Tuning

### 4A. Adaptive relay chunk sizing
- **File:** `src/intro_claw.rs` ~L856-865
- **Change:** Call `get_optimal_chunk_size()` for relay connections too, starting at 256KB, scaling down if throughput observations show relay is slow.

### 4B. Reduce stall watchdog: 8s → 5s for relay
- **Line:** ~L468
- **Why:** With push model, stalls should be detected faster.

### 4C. Increase DB flush cap: 50 → 100 per tick
- **Lines:** ~L1147, ~L1970, ~L2085, ~L2802
- **Why:** Faster recovery after reconnection with large pending backlog.

## Verification Plan

### After Phase 1 (constants):
1. **Same WiFi, 1:1:** Large file transfer. Confirm completes at full speed, no regression.
2. **Cross-network (Android VPN + iOS WiFi):** 1:1 file send from Android. Confirm transfer completes (not stuck at 0%).
3. **Cross-network group:** Send file to group with 2+ members on different networks. Confirm all receive.
4. **Monitor relay server:** Check RBN CPU/memory/bandwidth during transfers.

### After Phase 2 (push model):
1. Same tests as Phase 1.
2. **Relay stability:** Send 5 files in sequence over relay. Confirm no stalls or zombie transfers.
3. **Push→pull fallback:** Kill relay circuit mid-transfer. Confirm stall watchdog recovers and transfer completes via pull.

### After Phase 3 (group optimization):
1. **Group fan-out:** Send 50MB file to group of 3 members on different networks. Confirm all receive within expected time.
2. **Parallel seeding:** Member A downloads file, then Member B starts download. Confirm B pulls from both sender and A.

## Backward Compatibility

All Phase 1 changes are sender-side tuning — no protocol changes. Old receivers work with new senders (chunk size is in the manifest). Old senders with new receivers also work (pull model still functions).

Phase 2 (push model) sends unsolicited chunks to receivers. Old receivers already handle this — `handle_file_chunk` processes any chunk regardless of whether it was explicitly requested (chunks are indexed, duplicates are harmless via `received_chunks` HashMap).

Phase 3 changes are internal optimizations — no wire protocol changes.
